1. 前言

万象是个好方案，特别是加入音调之后，非常符合直觉，都不用拆字反查了。但是由于万象方案覆盖面很广，其实很多东西都是没必要的，添加了启用了反而可能影响性能。这里跟大家分享一下我自用的万象裁切方案，其中关闭了很多功能，但启用了音调输入和时间输入功能，性能比全启用的万象好一点。

3. wanxiang_pro.schema.yaml

我没有采用custom的方式，而是直接修改pro文件，再在每次更新时将当前文件和新文件diff一下，比对是否有额外修改。如果有，则修改当前文件，然后将下载下来的方案中除pro-schema外的其他文件覆盖rime文件夹内文件。

开头修改了一下说明，加入版本号以便我自己知道从哪个版本的万象方案里修改而来。

删除了mixcode、english和reverse方案，不需要英语单词、混合输入与反查。

# Rime schema
# encoding: utf-8
# 方案说明

  version: "LTS"
  author:
    - amzxyz
  description: |
    高度裁切的万象拼音，15.9.9

保留了几个开关，默认开启了emoji

switches:
  - name: full_shape                   #全角、半角字符输出
    states: [ 半角, 全角 ]
  - name: emoji                        #候选出现emoji滤镜，会显示在相应的候选后面，万象侧重于tips提示，避免候选被占用，因此默认为reset: 0，归属opencc emoji滤镜
    states: [ 表情关, 表情开 ]
    reset: 1

默认关闭了声调转换

  - name: prediction
    states: [ 预测关, 预测开 ]            # 这个名称能用于插件也能用于万象的lua实现，用来开启关闭预测功能
  - options: [ raw_input, tone_display, full_pinyin ]   #开启后在输入编码的位置实时转换为带声调全拼或者不带声调全拼，不开启则采用系统配置原始编码，影响的是preedit_format,归属：super_preedit.lua
    states: [ 原编码, 有声调, 无声调 ]
    reset: 0     #对于开关组从0开始数，第几个就reset几可设为默认
  - options: [ s2s, s2t, s2hk, s2tw ]  # 简繁转换开关组，可以在一个空选项和多个实际“- simplifier@s2hk”引入的项目之前切换，这是一个开关组，你可以将其中任意一个s2s等设置为toggle快捷键，多次按下将轮询
    states: [ 简体, 通繁, 港繁, 臺繁 ]

关闭了简码，不需要的功能

  - name: abbrev
    states: [ 简码关, 简码开 ]
    reset: 0
  - name: char_priority                #多体现在编码重合但候选有单字或者多字的情况`引导的辅码查词时是否单字优先，全拼常见，类似于特定编码情况下、反查状态下的调序能力。归属：super_lookup.lua
    states: [词组先, 单字先]

6. 附言：为什么移除lua脚本

我把整套lua脚本发给deepseek，让其帮我看看哪些最可能导致weasel不停dump，这是它给我的回答：

针对你提供的整套万象拼音 Lua 脚本，经过逐文件审查，以下几份 Lua 最有可能导致 Weasel 不停生成 dump 文件或频繁崩溃：

1.shijian.lua — 极高嫌疑

死循环风险：nl_shengri2 函数中的 while nl_shengri(y, m, d) == -1 do y = math.floor(y + 1) end
当 nl_shengri 因内部逻辑错误（如日期转换异常）始终返回 -1 时，该循环会无限递增年份，耗尽 Lua 整数或进入超长时间计算，导致 Weasel 进程卡死，被系统强制终止生成 dump。

触发场景：任何需要查询农历节日、节气或 /jr /day 等命令的输入，都会调用此函数。

建议修复：添加最大年份阈值（如当前年+100），超过即退出循环并返回无效提示。

2.super_lookup.lua — 高嫌疑

递归溢栈：match_fuzzy_recursive 函数使用深度递归进行模糊匹配，且未限制递归深度。当用户进行复杂反查操作（如输入较长辅码）时，递归层级可能瞬间突破 Lua 栈限制（默认约 200 层），直接引发栈溢出崩溃。

频繁调用：该函数在反查过滤阶段 func 中被调用，每次按键都会执行，任何异常输入都可能触发崩溃。

建议修复：改为非递归算法（如迭代或动态规划），或加入递归深度限制并提前终止。

3.super_calculator.lua — 中度嫌疑

潜在死循环：24 点计算 solve24 中嵌套多层循环，以及解方程函数（如 solveQuarticEquation）存在复杂的条件分支和大量运算。虽然单独执行不太会崩溃，但若与 V 键模式持续交互，可能导致超时或内存压力过大，引发间接崩溃。

高频调用：计算器在 V 模式下每次按键都会触发完整表达式求值，若表达式异常可能产生意想不到的死锁。

建议修复：为关键循环添加执行时间或迭代次数上限，采用尾调用优化或提前退出。

4.auto_phrase.lua / user_predict.lua — 低中度嫌疑

DB 操作异常：频繁操作 LevelDB 用户词典（Memory:update_userdict），若用户词典文件损坏或读写冲突，可能导致底层 C++ 抛出异常，触发 dump。

user_predict.lua 的清理命令：/clean 时全量遍历 DB 并批量删除，数据量大时可能占用过多 CPU / IO，间接影响稳定性。

建议修复：为数据库操作增加 .pcall 保护，并限制单次操作数量。

我并未用过农历或节气的日期计算功能，因此 shijian.lua 不是罪魁祸首。又因为我很少使用反查和计算器功能，所以取消了相应的lua。
